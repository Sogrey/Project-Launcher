# 本地项目启动器 (Project Launcher) 产品需求文档 (PRD)

| 文档版本 | V1.0 |
| :--- | :--- |
| **项目名称** | Project Launcher / 启航 (暂定) |
| **目标用户** | 经常在本地需要同时启动多个 Vue (Node) 等项目的全栈/前端开发人员 |
| **技术栈** | 前端 (Vue 3 + TypeScript) + 核心后台 (Rust / Tauri) + 进程通信 (Tauri IPC) |
| **UI 风格参考** | 深色极简模式、卡片看板、数据可视化 |

---

## 一、 核心项目目标
*   **痛点终结：** 告别在多个 VSCode/IDE 窗口间来回切换寻找终端的历史。
*   **一键管理：** 提供可视化的控制面板，对本地项目实现**一键扫描、一键启动、一键停止、随时查看日志**。
*   **不依赖 IDE：** 项目启动器独立于代码编辑器运行，作为常驻后台的“本地服务器管家”。

## 二、 功能需求详述 (Functional Requirements)

### 1. 工作区与项目库管理 (自动发现能力)
*   **1.1 工作区设置：**
    *   用户通过 UI 顶部的设置入口或初始引导页，选择一个**本地根目录**（如 `D:\MyProjects`）。
    *   支持 1 个工作区设定，工作区路径保存在本地配置文件中。
*   **1.2 自动扫描与识别：**
    *   **递归扫描：** 应用自动遍历该工作区下的所有子目录。
    *   **技术栈判定：** 
        *   通过识别根目录下是否存在 `package.json` 判定为 Node.js 项目（**V1 核心支撑**）。
        *   预埋扩展点：为后续识别 `pom.xml` (Java)、`go.mod` (Go) 做准备。
*   **1.3 智能命令提取 (关键交互)：**
    *   **解析 Scripts：** 自动解析 `package.json` 中的 `scripts` 字段。
    *   **智能推荐：** 自动将 `dev`、`serve`、`start`、`build` 设置为推荐的启动命令。若匹配到多个（如 `dev:client`, `dev:server`），进行列表罗列。
    *   **默认命令选择：** 自动选择 `dev` 为默认启动脚本（开发者可手动切换）。
*   **1.4 项目配置自定义：**
    *   **重命名别名：** 用户可为扫描出的项目自定义显示名称（避免文件夹名和项目名不一致）。
    *   **端口映射显示：** 用户可手动配置项目启动后的默认端口（如 5173, 3000），方便在卡片上直接展示 `http://localhost:3000` 一键跳转链接。
    *   **环境变量注入：** 支持针对特定项目配置启动前置环境变量（例如：`NODE_ENV=production` 或 `PORT=8080`）。

### 2. 看板控制台 (可视化核心)
*   **2.1 全局统计 (Dashboard)：**
    *   位于看板上方（参考截图顶部三列）。显示：总项目数、当前运行数、今日启动次数。
*   **2.2 卡片看板布局 (核心交互区)：**
    *   采用**多列瀑布流**布局，分为三个核心状态列：
        *   ⚪ **待处理 (Stopped)：** 识别出但未启动的项目列表。
        *   🔵 **进行中 (Running)：** 当前正在后台运行的项目列表。
        *   ⚠️ **异常/挂起 (Errored)：** 启动失败或意外中途退出的项目（需高亮显示）。
*   **2.3 项目卡片交互：**
    *   每张卡片必须包含以下元素：
        1.  **项目名称**（或自定义别名）。
        2.  **当前状态**（等待/运行中/失败），并配有对应的状态颜色/图标。
        3.  **启动命令选择器** (Dropdown): 允许在启动前快速切换脚本（如 `dev` -> `build`）。
        4.  **核心操作按钮：** `启动` / `停止` / `重启` (重启 = 先停止再启动)。
        5.  **日志入口：** 点击卡片或卡片上的特定图标，**展开/弹出**日志视图。

### 3. 实时日志与终端模拟 (对标 VS Code 体验 - 高优需求)
*   **3.1 嵌入式/弹出式终端：**
    *   支持在当前项目卡片下方**展开一个日志面板**，或点击后**弹出一个独立的日志窗口**。
*   **3.2 全色彩日志渲染：**
    *   **背景明确：** 必须强制在 Rust 层注入环境变量 `FORCE_COLOR=1` 来强制终端输出 ANSI 颜色码。
    *   **渲染引擎：** 前端不能采用纯文本方式拼接（极其卡顿），必须使用 **`xterm.js`** 作为底层渲染引擎，完美解析颜色、高亮、光标移动和报错信息。
*   **3.3 日志流式传输与防卡顿机制：**
    *   **Rust 缓冲区：** Rust 层不采用“读到一行发一行”的逻辑，而是采用 **时间窗口缓冲（如 200ms 为一个周期）** 或 **累计 100 行发一次** 的策略，降低 Tauri IPC 通信频率。
    *   **高吞吐支持：** 保证在 `npm run build` 这种瞬间输出上千行日志的场景下，前端 UI 不卡顿。
*   **3.4 日志快捷键与辅助功能：**
    *   提供“清空当前日志”按钮。
    *   提供“自动滚动到底部”（跟随输出）的开关。
    *   **查错辅助：** 允许在日志面板中搜索关键字（如 `Error`, `Failed`）。

### 4. 进程守护与系统级集成 (稳定性和易用性)
*   **4.1 进程生命周期管理：**
    *   **正确杀死：** 必须确保停止时能正确 Kill 掉包括子进程在内的整个进程树（避免 `npm run dev` 停止后，`vite` 或 `webpack` 子进程依然卡住 3000 端口）。
*   **4.2 开机自启与托盘：**
    *   **配置项：** 支持开启/关闭“开机自启”。
    *   **系统托盘：** 关闭主窗口时，应用**不退出**，而是**最小化到系统托盘**。托盘右键菜单包含：显示主窗口、一键停止所有服务、退出。
*   **4.3 一键全停 (救命功能)：**
    *   在主界面看板下方或右上角，放置“一键停止所有项目”按钮，点击后强制终止所有正在运行的子进程。
*   **4.4 重启后恢复机制：**
    *   应用重启后，自动从本地 SQLite 或 JSON 配置文件中加载之前的项目列表。
    *   *(V2 功能)* 可设定应用重启后，是否自动重新启动之前处于“运行中”状态的项目。

## 三、 技术架构需求 (Technical Requirements)

*   **后端框架：** **Tauri V2** (利用 Rust 提供的系统级底层操作能力和安全性，替代 Electron)。
*   **进程管理库：** Rust 标准库 `std::process::Command` 或 `tokio::process` (异步非阻塞)。
*   **前端框架：** **Vue 3 + Vite + TypeScript**，状态管理推荐 Pinia。
*   **UI 组件库：** 推荐 `Element Plus` (深色模式) 或 `Ant Design Vue` 结合自定义 CSS，还原截图中的半透明毛玻璃效果。
*   **终端组件：** **`xterm.js`** (用于日志渲染) + `xterm-addon-fit` (自适应容器大小)。
*   **数据持久化：** `tauri-plugin-store` (使用 JSON 文件存储配置，轻量级)。

## 四、 数据流与核心伪代码逻辑 (供开发参考)

1.  **扫描事件：** Vue 调用 Rust Tauri Command `scan_directory(path)`。
2.  **Rust 返回结果：** 返回一个 JSON 数组：`[{ name: "project-a", path: "D:/...", scripts: ["dev", "build"] }]`，Vue 渲染成待处理列表。
3.  **启动事件：** Vue 调用 Rust Tauri Command `start_project(path, script_name)`。
4.  **Rust 进程管理与通信 (关键)：**
    ```rust
    // Rust 伪代码
    let (mut tx, mut rx) = channel::<String>();
    let mut child = tokio::process::Command::new("npm")
        .args(["run", script_name])
        .current_dir(path)
        .env("FORCE_COLOR", "1") // 强制颜色
        .stdout(std::process::Stdio::piped())
        .stderr(std::process::Stdio::piped())
        .spawn()?;
    
    // 每 200ms 读取一次 stdout 并批量发送到前端
    while let Some(line) = read_from_pipe().await {
        logs_buffer.push(line);
        if logs_buffer.len() > 50 || timer.elapsed() > 200ms {
            tx.send(logs_buffer.join("\n")).await?;
            logs_buffer.clear();
        }
    }
    ```
5.  **前端渲染：** Vue 监听 Tauri IPC Event `project:<id>:log`，拿到字符串数组，直接 `term.writeln(data)` 喂给 xterm.js。

## 五、 非功能性需求 (NFR)

*   **性能：** 启动器主界面内存占用 < 200MB。启动 10+ 个 Vue 项目时 UI 无明显卡顿。
*   **稳定性：** 极端情况（如主窗口崩溃），子进程不能被挂在后台变成僵尸进程。
*   **响应速度：** 在点击“停止”到端口真正释放，控制在 2 秒以内。
*   **用户引导：** 对于第一次使用的用户，需提供“选择工作区”的清晰交互引导。

## 六、 版本迭代计划

*   **MVP 版本 (V1.0):**
    *   支持选择 1 个工作区。
    *   仅支持 Node.js (Vue) 项目，能自动识别和选择 `scripts`。
    *   实现基础看板、启动/停止、重启。
    *   **实现带颜色的实时日志流 (`xterm.js`)。**
    *   支持系统托盘常驻、一键全停。
*   **进阶版 (V2.0):**
    *   支持 Python (Flask/Django)、Go 项目的识别和启动。
    *   支持端口跳转打开浏览器。
    *   支持对单个项目配置特定的环境变量。
    *   增加日志搜素和“错误高亮”功能。
    *   开机自启功能开关。
